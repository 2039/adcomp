3 Tutorial {#Tutorial}
4 ========
5 
6 A TMB project consists of an R file (*.R) and a C++ file (*.cpp). The
7 R file does pre- and post processing of data in addition to maximizing
8 the log-likelihood contained in *.cpp. See [Examples](#Examples) for more
9 details. All R functions are documented within the standard help
10 system in R. This tutorial describes how to write the C++ file, and
11 assumes familiarity with C++ and to some extent with R.
12 
13 The purpose of the C++ program is to evaluate the objective function,
14 i.e. the negative log-likelihood of the model. The program is compiled
15 and called from R, where it can be fed to a function minimizer like
16 `nlminb()`.
17 
18 The objective function should be of the following C++ type:
19 
20 
21 ```{cpp}
22 #include <TMB.hpp>
23 
24 template<class Type>
25 Type objective_function<Type>::operator() ()
26 {
27 .... Here goes your C++ code .....
28 }
29 ```
30 
31 The first line includes the source code for the whole TMB package (and
32 all its dependencies). The objective function is a templated class
33 where `<Type>` is the data type of both the input values and
34 the return value of the objective function. This allows us to
35 evaluate both the objective function and its derivatives using the
36 same chunk of C++ code (via the AD package
37 [CppAD](http://www.coin-or.org/CppAD/Doc/doxydoc/html/index.html)). The
38 technical aspects of this are hidden from the user. There is however
39 one aspect that surprises the new TMB user. When a constant like "1.2"
40 is used in a calculation that affects the return value it must be
41 "cast" to Type:
42 
43 
44 ```{cpp}
45 Type nll; // Define variable that holds the return value (neg. log. lik)
46 nll = Type(1.2); // Assign value 1.2; a cast is needed.
47 ```
48 
49 ## Obtaining data and parameter values from R
50 Obviously, we will need to pass both data and parameter values to the
51 objective function. This is done through a set of macros that TMB
52 defines for us. To see which macros are available start typing
53 `DATA_` or `PARAMETER_` in the Doxygen search field of
54 your browser (you may need to refresh the browser window between each
55 time you make a new search). A simple example if you want to read a
56 vector of numbers (doubles) is the following
57 
58 
59 ```{cpp}
60 DATA_VECTOR(x); // Vector x(0),x(1),...,x(n-1), where n is the length of x
61 ```
62 
63 Note that all vectors and matrices in TMB uses a **zero-based**
64 indexing scheme. It is not necessary to explicitly pass the dimension
65 of `x`, as it can be retrieved inside the C++ program:
66 
67 
68 ```{cpp}
69 int n = x.size();
70 ```
71 
72 ## An extended C++ language
73 TMB extends C++ with functionality that is important for formulating
74 likelihood functions. You have different toolboxes available:
75 
76 - Standard C++ used for infrastructure like loops etc.
77 - Vector, matrix and array library (see [Matrices and arrays](#matrix_arrays))
78 - Probability distributions (see [Densities](#Densities) and [R style distributions](#R_style_distribution))
79 
80 In addition to the variables defined through the `DATA_` or
81 `PARAMETER_` macros there can be "local" variables, for which
82 ordinary C++ scoping rules apply. There must also be a variable that
83 holds the return value (neg. log. likelihood).
84 
85 
86 ```{cpp}
87 DATA_VECTOR(x); // Vector x(0), x(1), ..., x(n-1)
88 Type tmp = x(1);
89 Type nll = tmp * tmp;
90 ```
91 
92 As in ordinary C++ local variable tmp must be assigned a value before
93 it can enter into a calculation.
94 
95 ## Statistical modelling
96 TMB can handle complex statistical problems with hierarchical
97 structure (latent random variables) and multiple data sources. Latent
98 random variables must be continuous (discrete distributions are not
99 handled). The `PARAMETER_` macros are used to pass two types
100 of parameters.
101 
102 - **Parameters**: to be estimated by maximum likelihood. These include
103  fixed effects and variance components in the mixed model
104  literature. They will also correspond to hyper parameters with
105  non-informative priors in the Bayesian literature.
106 - **Latent random variables**: to be integrated out of the likelihood
107  using a Laplace approximation.
108 
109 Which of these are chosen is controlled from R, via the
110 `random` argument to the function `MakeADFun`. However,
111 on the C++ side it is usually necessary to assign a probability
112 distribution to the parameter.
113 
114 The purpose of the C++ program is to calculate the (negative) joint
115 density of data and latent random variables. Each datum and individual
116 latent random effect gives a contribution to log likelihood, which may
117 be though of as a "distribution assignment" by users familiar with
118 software in the BUGS family.
119 
120 
121 ```{cpp}
122 PARAMETER_VECTOR(u); // Latent random variable
123 Type nll = Type(0); // Return value
124 nll -= dnorm(u(0),0,1,true) // Distributional assignment: u(0) ~ N(0,1)
125 ```
126 
127 The following rules apply:
128 
129 - Distribution assignments do not need to take place before the latent
130  variable is used in a calculation.
131 - More complicated distributional assignments are allowed, say
132  *u(0)-u(1) ~ N(0,1)*, but this requires the user to have a deeper
133  understanding of the probabilistic aspects of the model.
134 - For latent variables only normal distributions should be used
135  (otherwise the Laplace approximation will perform poorly). For
136  response variables all probability distributions (discrete or
137  continuous) are allowed. If a non-gaussian latent is needed the
138  "transformation trick" can be used.
139 - The namespaces [R style distributions](#R_style_distribution) and [Densities](#Densities) contain
140  many probability distributions, including multivariate normal
141  distributions. For probability distributions not available from
142  these libraries, the user can use raw C++ code:
143 
144  ```{cpp}
145  DATA_VECTOR(y); // Data vector
146  Type nll = Type(0); // Return value
147  nll -= sum(-log(Type(1.0)+y*y)); // y are i.i.d. Cauchy distributed
148  ```
149 
150 See [Toolbox](#Toolbox) for more about statistical modelling.

167 Matrices and arrays {#matrix_arrays}
168 ===================
169 
170 ## Relationship to R
171 In R you can apply both matrix multiplication (`%*%`) and elementwise
172 multiplication (`*`) to objects of type "matrix", i.e. it is the
173 operator that determines the operation. In TMB we instead have two
174 different types of objects, while the multiplication operator (`*`) is
175 the same:
176 
177 - `matrix`: linear algebra
178 - `array`: elementwise operations; () and [] style indexing.
179 - `vector`: can be used in linear algebra with `matrix`, but at the
180  same time admits R style element-wise operations.
181 
182 See the file \ref matrix_arrays.cpp for examples of use.
183 
184 ## Relationship to Eigen
185 The TMB types `matrix` and `array` (in dimension 2) inherits
186 directly from the the Eigen types Matrix and Array. The advanced user
187 of TMB will benefit from familiarity with the
188 [Eigen documentation](http://eigen.tuxfamily.org/dox/group__TutorialMatrixClass.html).
189 Note that arrays of dimension 3 or higher are specially implemented in
190 TMB, i.e. are not directly inherited from Eigen.

193 R style probability distributions {#R_style_distribution}
194 =================================
195 
196 Attempts have been made to make the interface (function name and
197 arguments) as close as possible to that of R.
198 
199 - The densities (`d...`) are provided both in the discrete and
200  continuous case, cumulative distributions (`p...`) and inverse
201  cumulative distributions (`q...`) are provided only for continuous
202  distributions.
203 - Scalar and `vector` arguments (in combination) are supported, but
204  not `array` or `matrix` arguments.
205 - The last argument (of type `int`) corresponds to the `log`
206  argument in R: 1=logaritm, 0=ordinary scale. `true` (logaritm) and
207  `false` (ordinary scale) can also be used.
208 - Vector arguments must all be of the same length (no recycling of
209  elements). If vectors of different lengths are used an "out of
210  range" error will occur, which can be picked up by the debugger.
211 - `DATA_IVECTOR()` and `DATA_INTEGER()` cannot be used with
212  probability distributions, except possibly for the last (log)
213  argument.
214 - An example:
215 
216  ```{cpp}
217  DATA_SCALAR(y);
218  DATA_VECTOR(x);
219  vector<Type> rate(10);
220  matrix<Type> rate_matrix(10, 10);
221  dexp(y, rate, true); // OK, returns vector of length 10 of log-densities
222  dexp(x, rate, true); // OK if x is length 10
223  dexp(x, rate_matrix, true); // Error, matrix arguments not allowed
224  ```
225 - To sum over elements in the vector returned use
226 
227  ```{cpp}
228  sum(dexp(x,rate));
229  ```

232 Multivariate distributions {#Densities}
233 ==========================
234 
235 The namespace
236 
237 
238 ```{cpp}
239 using namespace density;
240 ```
241 
242 gives access to a variety of multivariate normal distributions:
243 
244 - Multivariate normal distributions specified via a covariance matrix
245  (structured or unstructured).
246 - Autoregressive (AR) processes.
247 - Gaussian Markov random fields (GMRF) defined on regular grids or
248  defined via a (sparse) precision matrix.
249 - Separable covariance functions, i.e. time-space separability.
250 
251 These seemingly unrelated concepts are all implemented via the notion
252 of a `distribution`, which explains why they are placed in the same
253 namespace. You can combine two `distributions`, and this lets you
254 build up complex multivariate distributions using extremely compact
255 notation. Due to the flexibility of the approach it is more abstract
256 than other parts of TMB, but here it will be explained from
257 scratch. Before looking at the different categories of multivariate
258 distributions we note the following which is of practical importance:
259 
260 - All members in the `density` namespace return the **negative** log
261  density, opposed to the univariate densities in [R style distributions](#R_style_distribution).
262 
263 ## Multivariate normal distributions
264 Consider a zero-mean multivariate normal distribution
265 with covariance matrix *Sigma* (symmetric positive definite),
266 that we want to evaluate at _x_:
267 
268 
269 ```{cpp}
270 int n = 10;
271 vector<Type> x(n); // Evaluation point
272 x.fill(0.0); // Point of evaluation: x = (0,0,...,0)
273 ```
274 
275 The negative log-normal density is evaluated as follows:
276 
277 
278 ```{cpp}
279 using namespace density;
280 matrix<Type> Sigma(n,n); // Covariance matrix
281 // ..... User must assign value to Sigma here
282 res = MVNORM(Sigma)(x); // Evaluate negative log likelihod
283 ```
284 
285 In the last line `MVNORM(Sigma)` should be interpreted as a
286 multivariate density, which via the last parenthesis `(x)` is
287 evaluated at `x`. A less compact way of expressing this is
288 
289 
290 ```{cpp}
291 MVNORM_t<Type> your_dmnorm(Sigma);
292 res = your_dmnorm(x);
293 ```
294 
295 in which `your_dmnorm` is a variable that holds the "density".
296 
297 _Sigma_ can be parameterized in different ways. Due to the symmetry of
298 *Sigma* there are at most *n(n+1)/2* free parameters (*n* variances
299 and *n(n-1)/2* correlation parameters). If you want to estimate all of
300 these freely (modulo the positive definite constraint) you can use
301 `UNSTRUCTURED_CORR()` to specify the correlation matrix, and
302 `VECSCALE()` to specify variances. `UNSTRUCTURED_CORR()` takes as
303 input a vector a dummy parameters that internally is used to build the
304 correlation matrix via its cholesky factor.
305 
306 
307 ```{cpp}
308 using namespace density;
309 int n = 10;
310 vector<Type> unconstrained_params(n*(n-1)/2); // Dummy parameterization of correlation matrix
311 vector<Type> sds(n); // Standard deviations
312 res = VECSCALE(UNSTRUCTURED_CORR(unconstrained_params),sds)(x);
313 ```
314 
315 If all elements of `dummy_params` are estimated we are in effect
316 estimating a full correlation matrix without any constraints on its
317 elements (except for the mandatory positive definiteness). The actual
318 value of the correlation matrix, but not the full covariance matrix,
319 can easily be assessed using the `.cov()` operator
320 
321 
322 ```{cpp}
323 matrix<Type> Sigma(n,n);
324 Sigma = UNSTRUCTURED_CORR(unconstrained_params).cov();
325 REPORT(Sigma); // Report back to R session
326 ```
327 
328 ## Autoregressive processes
329 Consider a stationary univariate Gaussian AR1 process
330 *x(t),t=0,...,n-1*. The stationary distribution is choosen so that:
331 
332 - *x(t)* has mean 0 and variance 1 (for all *t*).
333 
334 The multivariate density of the vector *x* can be evaluated as follows
335 
336 
337 ```{cpp}
338 int n = 10;
339 using namespace density;
340 
341 vector<Type> x(n); // Evaluation point
342 x.fill(0.0); // Point of evaluation: x = (0,0,...,0)
343 Type rho = 0.2; // Correlation parameter
344 res = AR1(rho)(x); // Evaluate negative log-density of AR1 process at point x
345 ```
346 
347 Due to the assumed stationarity the correlation parameter must
348 satisfy:
349 
350 - **Stationarity** constraint: -1 < *rho* < 1
351 
352 Note that *cor[x(t),x(t-1)] = rho*.
353 
354 The `SCALE()` function can be used to set the standard deviation.
355 
356 
357 ```{cpp}
358 Type sigma = 2.1; // standard deviation of x
359 res = SCALE(AR1(rho),sigma)(x);
360 ```
361 
362 Now, *var[x(t)] = sigma^2*. Because all elements of `x` are scaled by
363 the same constant we use SCALE rather than VECSCALE.
364 
365 #### Multivariate AR1 processes
366 This is the first real illustrations of how distributions can be used
367 as building blocks to obtain more complex distributions. Consider the
368 *p* dimensional AR1 process
369 
370 
371 ```{cpp}
372 int n = 10; // Number of time steps
373 int p=3; // dim(x)
374 array<Type> x(p,n); // Evaluation point
375 ```
376 
377 The columns in `x` refer to the different time points. We then
378 evaluate the (negative log) joint density of the time series.
379 
380 
381 ```{cpp}
382 MVNORM_t<Type> your_dmnorm(Sigma); // Density of x(t)
383 Type phi; // Correlation parameter
384 res = AR1(phi,your_dmnorm)(x);
385 ```
386 
387 Note the following:
388 
389 - We have introduced an intermediate variable `your_dmnorm`, which
390  holds the p-dim density marginal density of *x(t)*. This is a
391  zero-mean normal density with covariance matrix `Sigma`.
392 - All *p* univarite time series have the same serial correlation *phi*.
393 - The multivariate process *x(t)* is stationary in the same sense as
394  the univariate AR1 process described above.
395 
396 #### Higher order AR processes
397 
398 There also exists \ref ARk_t of arbitrary autoregressive order.
399 
400 ## Gaussian Markov random fields (GMRF)
401 GMRF may be defined in two ways:
402 
403 1. Via a (sparse) precision matrix Q.
404 2. Via a d-dimensional lattice.
405 
406 For further details please see \ref GMRF_t. Under 1) a sparse Q
407 corresponding to a Matern covariance function can be obtained via the
408 \ref R_inla namespace.
409 
410 ## Separable construction of covariance (precision) matrices
411 A typical use of separability is to create space-time models with a
412 sparse precision matrix. Details are given in \ref SEPARABLE_t. Here
413 we give a simple example.
414 
415 Assume that we study a quantity `x` that changes both in space and
416 time. For simplicity we consider only a one-dimensional space. We
417 discretize space and time using equidistant grids, and assume that the
418 distance between grid points is 1 in both dimensions. We then define
419 an `AR1(rho_s)` process in space and one in time `AR1(rho_t)`. The
420 separable assumption is that two points `x1` and `x2`, separated in
421 space by a distance `ds` and in time by a distance `dt`, have
422 correlation given by
423 
424 `rho_s^ds*rho_t^dt`
425 
426 This is implemented as
427 
428 
429 ```{cpp}
430 using namespace density;
431 int n_s = 10; // Number of grid points in space
432 int n_t = 10; // Number of grid points in time
433 Type rho_s = 0.2; // Correlation in space
434 Type rho_t = 0.4; // Correlation in time
435 
436 array<Type> x(n_s,n_t);
437 x.setZero(); // x = 0
438 
439 res = SEPARABLE(AR1(rho_t),AR1(rho_s))(x);
440 ```
441 
442 Note that the arguments to `SEPARABLE()` are given in the opposite order
443 to the dimensions of `x`.

446 Example collection {#Examples}
447 ==================
448 
449 - A list of all examples is found on the "Examples" tab on the top of the page.
450 - Locations of example files: `adcomp/tmb_examples` and `adcomp/TMB/inst/examples`.
451 - For each example there is both a `.cpp` **and** a `.R` file. Take for instance the linear regression example:
452 - **C++ template**
453 
454  ```{cpp}
455  // Simple linear regression.
456  #include <TMB.hpp>
457  template<class Type>
458  Type objective_function<Type>::operator() ()
459  {
460  DATA_VECTOR(Y);
461  DATA_VECTOR(x);
462  PARAMETER(a);
463  PARAMETER(b);
464  PARAMETER(logSigma);
465  ADREPORT(exp(2*logSigma));
466  Type nll = -sum(dnorm(Y, a+b*x, exp(logSigma), true));
467  return nll;
468  }
469 
470  ```
471 
472 - **Controlling R code**
473 
474  ```r
475  library(TMB)
476  compile("linreg.cpp")
477  dyn.load(dynlib("linreg"))
478  set.seed(123)
479  data <- list(Y = rnorm(10) + 1:10, x=1:10)
480  parameters <- list(a=0, b=0, logSigma=0)
481  obj <- MakeADFun(data, parameters, DLL="linreg")
482  obj$hessian <- TRUE
483  opt <- do.call("optim", obj)
484  opt
485  opt$hessian ## <-- FD hessian from optim
486  obj$he() ## <-- Analytical hessian
487  sdreport(obj)
488  ```
489 
490 - To run this example use the R command
491 
492  ```r
493  source("linreg.R")
494  ```

531 Compilation and run time errors {#Errors}
532 ===============================
533 
534 The R interface to the debugger (gdb) is documented as part of the R
535 help system, i.e. you can type `?gdbsource` in R to get info. The
536 current document only adresses isses that the relate to C++.
537 
538 ## Compilation errors
539 It may be hard to understand the compilation errors for the following
540 reasons
541 
542 - The Eigen libraries use templated C++ which generate non-intuitive
543  error messages.
544 
545 ## Run time errors
546 Run time errors are broadly speaking of two types:
547 
548 - Out-of-bounds (you are "walking out of an array")
549 - Floating point exceptions
550 
551 You can use the debugger to locate both types of errors, but the
552 procedure is a little bit different in the two cases. The following
553 assumes that you have the GNU debugger `gdb` installed.
554 
555 ### Out-of-bounds error
556 An example is:
557 
558 
559 ```{cpp}
560 vector<Type> y(4);
561 y(5); // 5 is not a valid index value here
562 ```
563 
564 This will cause TMB and R to crash with the following error message:
565 
566 > TMB has received an error from Eigen. The following condition was not met:
567 > index >= 0 && index < size()
568 > Please check your matrix-vector bounds etc., or run your program through a debugger.
569 > Aborted (core dumped)
570 
571 So, you must restart R and give the commands
572 
573 
574 ```r
575 library(TMB)
576 gdbsource("my_project.R")
577 ```
578 
579 > \#5 objective_function<double>::operator() (this=<optimized out>) at nan_error_ex.cpp:11
580 
581 and you can see that the debugger points to line number 11 in the .cpp
582 file. `gdbsource()` is an R function that is part of TMB.
583 
584 ### Floating point exception
585 If you on the other hand perform an illegal mathematical operation,
586 such as
587 
588 
589 ```{cpp}
590 Type f = sqrt(-1.);
591 ```
592 
593 R will not crash, but the objective function will return a NaN value.
594 However, you will not know in which part of your C++ code the error
595 occured. By including the `fenv.h` library (part of many C++
596 compilers, but can otherwise be downloaded from
597 http://www.scs.stanford.edu/histar/src/uinc/fenv.h)
598 
599 `nan_error_ex.cpp`:
600 
601 ```{cpp}
602 // Illustrates how to make the debugger catch a floating point error.
603 #include <TMB.hpp>
604 #include <fenv.h> // Extra line needed
605 
606 template<class Type>
607 Type objective_function<Type>::operator() ()
608 {
609  feenableexcept(FE_INVALID | FE_OVERFLOW | FE_DIVBYZERO | FE_UNDERFLOW); // Extra line needed
610 
611  DATA_SCALAR(lambda);
612  PARAMETER(x);
613  Type f;
614 
615  f = sqrt(-1.); // FE_INVALID (sqrt(-1.) returns NaN)
616  //f = 1./0.; // FE_DIVBYZERO (division by zero)
617  //f = exp(100000.); // FE_OVERFLOW (exp(100000.) returns Inf) [Does not work on all platforms]
618  //f = exp(-100000.); // FE_UNDERFLOW (exp(-100000.) returns 0)
619  return f;
620 }
621 ```
622 
623 
624 a floating point exception will be turned into an actual error that
625 can be picked up by the debugger. There are only two extra lines that
626 need to be included ("//Extra line needed" in the above example).
627 
628 When we try to run this program in the usual way, the program crashes:
629 
630 
631 ```r
632 source("nan_error_ex.R")
633 ```
634 
635 > Floating point exception (core dumped)
636 > tmp3>
637 
638 At this stage you should run the debugger to find out that the
639 floating point exception occurs at line number 14:
640 
641 
642 ```r
643 library(TMB)
644 gdbsource("nan_error_ex.R")

652 Toolbox {#Toolbox}
653 =======
654 
655 First read the *Statistical Modelling* section of [Tutorial](#Tutorial).
656 
657 ### Non-normal latent variables (random effects)
658 
659 The underlying latent random variables in TMB must be Gaussian for the
660 Laplace approximation to be accurate. To obtain other distributions,
661 say a gamma distribution, the "transformation trick" can be used. We
662 start out with normally distributed variables `u` and transform these
663 into new variables `w` via the `pnorm` and `qgamma` functions as
664 follows:
665 
666 
667 ```{cpp}
668 PARAMETER_VECTOR(u); // Underlying latent random variables
669 Type nll=Type(0.0);
670 nll -= sum(dnorm(u,Type(0),Type(1),true)); // Assign N(0,1) distribution u
671 vector<Type> v = pnorm(u,Type(0),Type(1),true); // Uniformly distributed variables (on [0,1])
672 vector<Type> w = qgamma(v,shape,scale,true);
673 ```
674 
675 `w` now has a gamma distribution.
676 
677 ### Discrete latent variables
678 
679 The Laplace approximation can not be applied to discrete latent
680 variables that occur in mixture models and HMMs (Hidden Markov
681 models). However, such likelihoods have analytic expressions, and may
682 be coded up in TMB. TMB would still calculate the exact gradient of
683 the HMM likelihood.
684 
685 ### Mixture models
686 
687 Although mixture models are a special case of discrete latent variable
688 models, they do deserve special attention. Consider the case that we
689 want a mixture of two zero-mean normal distributions (with different
690 standard deviations). This can be implemented as:
691 
692 
693 ```{cpp}
694 DATA_VECTOR(x);
695 PARAMETER_VECTOR(sigma); // sigma0 og sigma1
696 PARAMETER(p); // Mixture proportion of model 0
697 Type nll=Type(0.0);
698 nll -= sum( log( p * dnorm(x, Type(0), sigma(0), false)
699  + (1.0-p) * dnorm(x, Type(0), sigma(1), false) ) );
700 ```
701 
702 ### Time series
703 Autoregressive (AR) processes may be implemented using the compact
704 notation of section [Densities](#Densities). The resulting AR process may be
705 applied both in the observational part and in the distribution of a
706 latent variable.
707 
708 Nonlinear time must be implemented from scratch, as in the example
709 \ref thetalog.cpp
710 
711 ### Spatial models
712 TMB has strong support for spatial model and space-time models via the
713 `GMRF()` and `SEPARABLE()` functions, and the notion of a
714 `distribution`. The reader is referred to section [Densities](#Densities) for
715 details and examples.

721 ### I know R but not C++
722 
723 Summary of how syntax differs between R and C++:
724 
725 
726 ```{cpp}
727  R code C++/TMB code
728 
729 Comments # // // Comment symbol
730 Constants 3.4 Type(3.4); // Explicit casting recommended in TMB
731 Scalar x = 5.2 Type x = Type(5.2); // Variables must have type
732 Arrays x = numeric(10) vector<Type> x(10); // C++ code here does NOT initialize to 0
733 Indexing x[1]+x[10] x(0)+x(9); // C++ indexing is zero-based
734 Loops for(i in 1:10) for(i=1;i<=10;i++) // Integer i must be declared in C++
735 Increments x[1] = x[1] + 3 x(0) += 3.0; // += -= *= /= incremental operators in C++
736 ```
737 
738 ### I know C++
739 TMB specific C++ include:
740 - You should not use `if(x)` statements where `x` is a `PARAMETER`,
741  or is derived from a variable of type `PARAMETER`. (It is OK to use
742  `if` on `DATA` types, however.) TMB will remove the `if(x)`
743  statement, so the code will produce unexpected results.
